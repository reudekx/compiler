1 - b 어떻게 구현할지

    커서를 2개 선언한다. bc (begin 커서) 와 fc (forawrd 커서)

    chunck를 하나 읽는다.

    커서를 진행시키면서 token을 식별하고 변수명이나 리터럴 값들을 symbol table에 저장한다.

    위 과정들을 반복하다 chunk의 끝에 다다르고 eof가 발견되지 않으면 다음 chunk를 읽는다.

    이때, fc보다 bc의 값이 더 큰 경우에는, bc가 이전 chunk를 가르키는 것을 의미하게 된다.

        그런데 chunk 크기보다 더 큰 lexeme이 입력될 가능성이 있으므로 bc가 어느 chunk를 가리키고 있는지 확인할 방법이 필요하다.

        (사용자가 잘못된 코드를 입력하는 것은 막을 도리가 없기 때문에..)


    또한 lexeme이 식별되었을 때 bc로부터 fc 까지 진행하며 character를 하나씩 반환할 함수가 필요하다.

        chunk에 걸쳐져 있는 문자들을 나머지 연산을 이용해 적절히 처리하도록 구현하자.

3. Bytecode

    파싱 과정에서 AST를 생성하는 것보다 일련의 Bytecode들을 생성하도록 하자.

        AST 트리 구조보단 선형적인 Bytecode 배열이 더 효율적일 듯
            
            정확한 이유를 생각해 봐야할까? 필요하면 검색을 해보자.

    하나의 Bytecode는 하나의 Operator와 하나 이상의 Operand들로 이루어져 있다.

    Value 스택에 대한 Push, Pop 연산이 필요하다.

    클래스든 함수든 결국 숫자 혹은 문자, boolean 등의 리터럴에 대한 연산으로 환원시킬 수 있다.

    또한 jump 등의 명령어도 필요하다. 가령 if / else 와 같은 조건문, and, or과 같은 논리 연산자를 처리하기 위해 필요하다.


    [함수의 정의 / 호출]

    이를 어떻게 처리할지 고민해봤다.

    먼저, 함수라는 것은 결국 일련의 바이트코드 목록으로 나타낼 수 있다.

    따라서 start 인덱스와 end 인덱스를 유지하고 있으면 된다.

    다음으로, 파라미터에 값을 어떻게 전달할지 생각해 봤다.

    내 생각에는,
        함수의 정의 시점에 일련의 'Load <파라미터명>' 이라는 Bytecode를 삽입한다.

        그리고 이후 함수의 호출 시점에는 일련의 'Store <파라미터명>' 이라는 Bytecode를 삽입한다.

        이후 'Call <함수명>'을 삽입한다.

        그러면, 나중에 실행할 때 Call 이후 함수의 정의로 jump 하게 되면
        
        'Load <파라미터명>' 이 실행될 때 직전에 Store가 이뤄졌으므로, Value Stack에 저장된 Argument 값을 자동으로 확인할 수 있다.

    
    [애초에 함수의 Argument 값을 어떻게 구함?]

    가령, 함수의 인자에는 표현식들이 쉼표로 구분되어 전달될 것이다.

    결국 표현식의 처리 결과로는 Value 스택에 각각 '하나의 값'이 Push될 것이다.

    따라서 함수 입장에선 필요한 인자 개수만큼 Value 스택에서 Pop 한 뒤,

    이를 '파라미터명' 변수에 Store 해주면 된다.



    [파싱은 어떻게 진행?]

    파싱테이블을 이용하든가, 재귀 함수 형태로 파싱을 진행할 수 있을 것이다.

    전자의 경우에는 파싱 과정에서 쓰일 스택을 하나 둬야할 듯.

    이 시점에 문법에 대한 정확한 명세가 필요하다.

    진행을 시뮬레이션 해보자. 물론 문법 정의가 불명확하기 때문에 대충 상상하여 서술

        1. Parse 스택에 'Program' 이라는 Nonterminal Token을 하나 Push 한다.

        2. 'Program'은 'Expression'으로 확장된다. (Program ::= Expression)
            파싱 테이블이 어떻게 구성될지는 모르겠지만 일단 Current Token이 어떻든 Expression으로 확장된다고 가정.

        3. Expression은 여러 경우의 수가 존재한다.
            가령,
                ID = Expression

                ID ( arg1 , arg2 , arg3 )

            위와 같이 할당식과 호출식 둘 중 하나로 분화될 수 있다.

        4. 다시 이어서 진행해보자.

            Parse 스택에 'Expression'이 들어있고, 현재 토큰은 ID이다.

            이떄 lookahead 토큰을 살펴보아야 한다.

            lookahead 토큰이 = 이면 할당식이고, ( 이면 호출식이다.

            =====================================================

            내가 착각했던 부분이 있다.

            Lexing 과정에서 ID는 하나의 토큰으로써 읽히지만,

            Parsing 과정에서는 Nonterminal 토큰으로 간주해야 한다.

            =====================================================

            또한 생각을 해보니 할당식은 right-associative이다.

            따라서 이를 처리할 방법을 생각해봐야 한다.

[LL(1) 파서에 대한 정확한 이해]

1. LL(1) 파서를 위해선, 문법에서 왼쪽 재귀가 제거되어야 한다.

    가령, 
        E -> E '+' T | T
        T -> int | '(' E ')'

    위와 같은 식에서 E는 무한히 확장된다.

    따라서

    E' 등의 추가 심볼을 도입해서 해결해야 한다.

    가령,
        E -> T E'
        E' -> '+' T E' | e
        T -> int | '(' E ')'

2. right-associative 문법은 어떠한지?

    E -> id '=' E | id

3. 할당 연산

    Assignment 문법 rule을 추가하게 되면,

    ID '=' Expression 과 같은 형태가 될텐데,

    이러면 lookahead 토큰이 2개 필요하다. (ID를 본 뒤, '=' 까지 봐야 식별이 가능하다.)

    왜냐하면 가령 CallExpression ::= ID '(' Parameters ')'  와의 구분이 불가능하기 때문이다.

    따라서 Assignment를 따로 정의하지 말고,

    '=' 연산자가 등장하면 Assignment 'Action'을 진행하도록 해야 한다.

    다만 이 경우 파싱 과정에서 ID = Expression 이라는 문법을 강제할 수 없다.

    좌변에 ID 대신 다른 것이 등장할 수 있고, 이는 실행 시점에 확인할 수 있다.


    정리하자면, LL(1)이 성립하려면 어떤 Nonterminal이 2개 이상의 rule로 분기될 때,

    각 rule의 FIRST 값이 달라야 한다.

    ======================================================================

    그런데 생각해보니, 위처럼 ID를 따로 분리하게 되면

    CallExpression에서도 ID가 제거될텐데, 그럼 '(' Parameters ')' 만이 남게 되고

    이는 연산자 우선순위를 높이기 위한 소괄호와 구분할 수 없게 된다.

    따라서 어쩔 수 없이 lookahead 토큰을 2개 사용하는 LL(2) 문법을 정의해야 한다.


[lookahead 어떻게 처리할지]

Lookahead[2] 배열을 선언.

CurLookIndex 변수를 선언.

look(1)을 호출하면 Lookahead[CurLookIndex] 를 반환.
look(2)를 호출하면 Lookahead[1 - CurLookIndex]를 반환.

다만 look(2)를 할 때 gettok을 언제 호출해야 하는지?

Token을 소모하는 consume 함수가 있을 것이다.

consume 호출 시 look(2)에 대응되는 토큰을 미리 읽는다.

이후 current index를 바꿔주면 된다.
