표현식 지향 언어의 문법에 대해 생각해보자.

모든 코드 블록은 최소한 하나의 반환값을 가져야 한다.

한 블록 내에 여러 표현식이 존재한다면, 마지막으로 실행된 표현식을 반환한다.

다만 반복문의 경우 마지막 표현식을 반환하는 것이 크게 의미가 없을 뿐더러,

컴파일러 입장에선 언제 반복이 종료될지 알 수 없기 때문에 break와 같은 키워드로 명시적으로 값을 반환하도록 해야 한다.

또한 코드 블록이 종료될 때 반환된 값을 유지하기 위한 hold와 같은 편의성 키워드를 생각해 볼 수 있다.

    -> 생각해봤는데 단점이 많은 것 같아서 안 넣는 게 맞는 것 같다.

그리고 함수에 return문이 없어도 마지막 표현식의 값을 명시적으로 반환하도록 할 수 있다.

-> 그렇게 하면 장점이 많음. 다만 표현식 값을 반환하지 않으려면 return nil을 명시해야 할 수도 있음.


이번엔, 표현식의 끝을 명시적으로 알려야 하는가? 에 대해 생각해보자.

    end

    위 코드는 void 값을 반환하는가? 아니면 z = b 라는 표현식의 결과값을 반환하는가?

    사실, 위와 같은 상황은 생기지 않을 것이다.

    따라서 굳이 따로 처리를 할 필요는 없을 것 같다. (end의 존재만으로 충분하다.)

명확히 알고 있는 것은 아니다.

1. 함수
    return이 없으면 마지막 요소를 반환할 것인지?

    가령 expression이 처리되고 나면 stack에 최종 값이 쌓여있을 것이다.
    만약 여러 expression이 처리되는 경우 다음 expression이 시작될 때 stack에서 마지막 값을 pop하면 된다.
    
    함수의 끝에 도달했을 때 -> 함수의 끝을 명시한 Bytecode를 삽입하기 전에

        만약 pop 연산을 명시하지 않으면 값이 남아있을 것이기 때문에, return이 없어도 마지막 표현식의 결과를
        반환하는 것은 쉬운 일이다.

    일단 그렇게 하도록 하자.

2. 반복문
    마찬가지로, 반복문이 종료될 때 명시적으로 pop을 하지 않으면 마지막 표현식의 결과가 stack에 남아있게 된다.
    이를 반환하도록 할 수 있다.

    또한 break를 통해 탈출과 동시에 명시적으로 반환하는 것도 가능하다.

3. 조건문
    
    역시나 stack의 마지막 값을 반환하도록 하면 된다.

5. 선언

[JUMP 위치를 어떻게 구할 것인지?]

현재 떠오른 방법

1. PASS를 2번 돌린다.
    처음에는 위치를 Label 형태로 표시하고,
    이후 PASS에서 실제 index로 바꿔준다.

2. 스택을 이용한다.
    가능할진 모르겠으나 어쨌든 문법 구조가 정의되어 있으니 적절히 스택을 이용하면 될 수도?

    예시를 통해 생각해보자.

    a = 0
    while a < 5 do
        a += 1
    ok
    print(a)

    먼저 위치를 알고 있다고 가정하고 Bytecode 배열을 작성해보자.

    00 LOAD_CONST 0
    01 STORE 'a'
    02 LOAD 'a' ---------------> LOOP START  -------------- PUSH 02
    03 LOAD_CONST 5
    04 COMP_LESS
    05 JUMP_IF_FALSE ( ) -------> LOOP EXIT   -------------- POP WHEN EXECUTED
    06 LOAD 'a'
    07 LOAD_CONST 1
    08 ADD
    09 STORE 'a'
    10 JUMP  ( )  ----------------> JUMP BACK TO LOOP -------- POP WHEN EXECUTED
    11 OK (== LOOP END의 자리 표시자)  ------> LOOP END   --------------- PUSH 11 
    12 STORE 'a'
    13 JUMP 2
    14 LOAD 'a'
    15 PRINT
    16 END

    컴파일 시 JUMP STACK에는 [LOOP_START (02), AFTER_LOOP(12)]가 들어있을 것이다.

    중간에 탈출을 하게 되면 AFTER_LOOP로 이동해야 하고,
    그게 아니라면 조건 체크를 위해 LOOP_START로 다시 이동해야 한다.

    이때 탈출

    근데 생각해보니 결국 2번의 패스를 가정하고 있다.

중간 점검:
    어차피 컴파일 최적화 등을 위해 패스를 2번 이상 진행해야 하고,
    다만 이 과정에서 점프 위치를 찾기 위해 스택을 이용하면 된다.

[Jump 바이트코드 재논의]

더블 패스로 돌려서 Jump 위치를 지정해주려고 했는데,
그냥 Jump Bytecode 자체를 stack에 집어넣고 싱글 패스로 처리하면 된다.

바이트코드 예시를 위에서 다시 가져왔다.

    a = 0
    while a < 5 do
        a += 1
    ok
    print(a)

    먼저 위치를 알고 있다고 가정하고 Bytecode 배열을 작성해보자.

    00 LOAD_CONST 0
    01 STORE 'a'
    02 LOAD 'a' ---------------> LOOP START  -------------- PUSH 02 (JUMP TARGET)
    03 LOAD_CONST 5
    04 COMP_LESS
    05 JUMP_IF_FALSE ( ) -------> LOOP EXIT   -------------- PUSH 05 (JUMP INDEX)
    06 LOAD 'a'
    07 LOAD_CONST 1
    08 ADD
    09 STORE 'a'
    10 JUMP  ( )  ----------------> JUMP BACK TO LOOP -------- PUSH 10 (JUMP INDEX)
    11 OK (== LOOP END의 자리 표시자)  ------> LOOP END   --------------- PUSH 11 (JUMP TARGET)
    12 STORE 'a'
    13 JUMP 2
    14 LOAD 'a'
    15 PRINT
    16 END

JUMP TARGET
[02,  11]
JUMP INDEX
[05, 10]

10에서는 02로 가야하고,
05에선 11로 가야한다.

언뜻 JUMP TARGET에서 FIFO로 인출해야 하는 것처럼 보일 수도 있는데

여러 JUMP 문들이 중첩되면 전체적으로는 LIFO 방식으로 처리해야 함.

애초에 2개 뿐이기 때문에 하나의 JUMP 문에 대해선 FIFO든 LIFO든 알아서 하면 됨.

가령 WHILE / FOR 문의 경우 2개의 쌍이 존재함.

-> WHILE문이 끝날 때 pop 하도록 함수를 예약하면 됨.
    -> Symbol로 정의해서 넣어서 Action으로서 실행되도록 해야 할 듯.

IF 문은?

가령 IF ... ELIF ... ELSE 의 형태로 나타날 때,

순서대로 만약 조건을 만족하지 못한다면 다음 조건의 위치로 점프하고,
조건을 만족한다면 해당 조건 뒤의 블럭을 실행한 뒤 IF 문 바깥으로 JUMP 해야 한다.

마지막 ELSE의 경우 JUMP없이 그냥 끝내면 된다.

IF가 1개 있고 ELIF가 N개 있다고 가정해보자.

    IF ( ) J

    J
    ELIF () J, T

    J
    ELIF () J, T

    J
    ELSE T

    T

    J가 총 2 * (N - 1)개,
    T가 N개 존재.

    여기서,
        조건이 실패했을 때 다음 조건으로 가는 J가 N - 1개 있고 이에 대응되는 T가 N - 1개 있다.
        또한 조건을 만족하고 블록을 실행한 뒤 IF문의 마지막으로 가는 J가 N - 1개 있고 이에 대응되는 T가 1개 있다.

    즉,
    [J J J J J J]
    [T T T T]

    처럼 스택이 구성되면

    T에서 1개를 POP해서 유지하고 -> 이를 T' 라고 하면, 
    [J J J J J J]
    [T T T]

    T에서 POP -> J에서 POP한 뒤 SET -> J에서 POP 한 뒤 T'을 SET -> 반복

    위 과정을 따라서 Jump 위치를 구해주면 된다.


논리 연산자의 경우에는?

AND / OR 

-> AND의 경우

    A and B

    JUMP_IF_ZERO A
    JUMP_IF_ZERO B
    
    위의 분기문들과의 차이점이라면, JUMP 후에 
    Value를 POP 하지 않는다는 것이다. (가령 IF문의 경우 condition으로 이용된 표현식의 결과를 check한 뒤 POP 해줘야 한다.)

[표현식의 결과는 언제 POP 되어야 하는지?]

가령 break나 return에 대해 생각해보자.

JUMP를 하는 것 이외에도 Value Stack에서 적절히 필요없는 결과를 POP 해저야 한다.

그러나 이를 특별히 따로 처리할 필요는 없어보인다.

-> 애초에 표현식의 결과가 언제 POP 되는지를 생각해보고 그에 따라 동일하게 처리하면 될 것 같다.

가령 1 + 2 * 3에 대해 생각해보자.

이를 Bytecode로 나타내면,

PUSH 1
PUSH 2
PUSH 3
POP
POP
MUL
POP
ADD

대략 위와 같은 결과가 될 것이다.

반면 표현식'문'의 경우에는,

가령

a = 1 + 2 * 3

b = 1 + 2 * 3

이 경우에는 하나의 문장이 종료된 후 해당 결과를 POP 해줄 필요가 있다.

    -> 여러 문장이 등장하는 것은 block밖에 없으므로, block이 재귀적으로 top_expr을 생성할 때
        POP을 먼저 진행하도록 하면 된다.


[변수 & 값 처리]

상수의 경우 바이트코드에 그대로 나타내면 된다.

변수의 경우는?

컴파일러의 역할을 생각해보자.

1. 변수명을 특정한 정수값으로 변환한다.

    내 생각에는 해싱을 통해 변수명을 해시값으로 변경하고, 해당 해시값에 대해 1, 2, 3 의 오름차순 순서를 대응시키면 될 듯.
    -> 이후 바이트코드에서는 변수명 대신 1, 2, 3을 사용하고, 이를 통해 stack의 위치에 접근할 수 있다.

2. 지역 변수의 개수를 센다.

    이를 통해 ALLOCATE 바이트코드를 생성할 수 있다.



생각해보니, 동적 프로그래밍 언어이므로 지역변수의 개수만으로 미리 필요한 스택 영역을 계산하기가 어려워보인다.

따라서 파이썬처럼 스택에는 레페런스나 함수 포인터 등만 저장하고 실제 데이터는 힙에 생성하는 게 맞을 것 같다.

클로저를 구현하기에도 편할 듯

    -> 이 경우 가비지 컬렉션 기능을 추가해야 한다.

    -> 또한 힙 메모리를 관리할 방법이 필요하다.

        -> free list를 통해 현재 사용하고 있지 않은 블록의 위치를 알 수 있도록 한다.

        또한 힙 블록을 가령 2의 거듭제곱의 크기를 배수로 하여 여러개를 연속적으로 이용할 수도 있도록 해야 한다.

        내 생각에는 이걸 빌트인으로 구현하는 게 아니라 라이브러리로 구현하도록 해도 될 것 같음.

            -> 그럼 어느정도의 기능까지 미리 구현해놓을 건지?

                가령, allocate()와 free()는 미리 구현해야 한다.

                가비지 컬렉터의 구현을 위해 사용 중인 힙 메모리를 추적할 방법을 제공해야 한다.
                    -> 사용 중이지 않은 메모리 주소를 저장하는 free list를 제공할 수 있다.
